# 抢票流程时间压缩分析

## 当前时间线（从开售点击到提交完成）

| 阶段 | 最佳 | 典型 | 最差 | 主要耗时 |
|------|------|------|------|----------|
| INIT | 30ms | 100ms | 150ms | 3×adb shell 设置动画 |
| WAIT | 0 | 0 | 0 | NTP校时已移除，纯等待 |
| BUY | 100ms | 200ms | 515ms | 点击+Activity轮询(30ms间隔) |
| SELECT_PRICE | 140ms | 400ms | 2s | 确保票档页+查找坐标+点击 |
| CONFIRM | 800ms | 2s | 4.5s | 进入订单页+找提交按钮+点击 |
| **总计** | **~1s** | **~2.7s** | **~7s** | |

---

## 可优化项（按收益排序）

### P0: CONFIRM 阶段提交按钮查找（收益 500ms-1.5s）

**文件**: `src/phase_machine.py` 第 780-794 行

当前逻辑：首次 `time.sleep(0.3)` → 循环 `d.dump_hierarchy()` 查找提交按钮（每次 300-700ms），直到 2.5s 超时。

**优化方案**：
1. 首次等待从 300ms 降到 150ms（Activity 已确认进入订单页，UI 渲染不需要 300ms）
2. 用 `d(textContains="立即提交").exists` 替代完整 `dump_hierarchy()`，单次 ~50ms vs ~500ms
3. 如果 exists=True，再用 `d(textContains="立即提交").click()` 直接点击，省去坐标查找

```python
# 优化前
time.sleep(0.3)
while time.time() - _find_start < 2.5:
    xml = self.d.dump_hierarchy()
    submit_coords = self._find_submit_btn_coords(xml)
    if submit_coords: break
    time.sleep(0.1)

# 优化后
time.sleep(0.15)
_start = time.time()
while time.time() - _start < 2.0:
    if self.d(textContains="立即提交").exists:
        # 直接获取坐标并点击，省去 dump_hierarchy
        el = self.d(textContains="立即提交")
        cx, cy = el.center
        self._submit_btn_coords = (cx, cy)
        break
    time.sleep(0.05)
```

**风险**: 低。u2 API 的 textContains 比 dump_hierarchy 快 10 倍。

### P1: CONFIRM 阶段硬编码等待（收益 300-600ms）

**文件**: `src/phase_machine.py`

当前有 3 处硬编码 `time.sleep(0.3)`：
- 第 783 行：进入订单页后等渲染（300ms）
- 第 804 行：找完按钮后等页面稳定（300ms）
- 第 811 行：点击提交后等响应（300ms）

**优化方案**：
- 第 783 行：降到 100ms（Activity 已确认在订单页，按钮渲染不需要 300ms）
- 第 804 行：降到 100ms（坐标已找到，不需要额外等待）
- 第 811 行：去掉或降到 50ms（点击后直接进入 DONE 阶段，无需等待）

```python
# 优化前
time.sleep(0.3)  # 第783行
...
time.sleep(0.3)  # 第804行
...
self.d.click(*submit_coords)
time.sleep(0.3)  # 第811行

# 优化后
time.sleep(0.1)  # 第783行
...
time.sleep(0.1)  # 第804行
...
self.d.click(*submit_coords)
# 第811行去掉，直接进入DONE
```

**风险**: 低。点击提交后不需要等待 UI 稳定。

---

### P2: `_ensure_price_page()` 减少 dump_hierarchy 调用（收益 200-500ms）

**文件**: `src/phase_machine.py` 第 229-313 行

当前逻辑：Activity 判断在票档页后，最多 5 次 `dump_hierarchy()` + 80ms 等待（检查 ¥ 标记）。

**优化方案**：
1. 减少重试次数从 5 到 3
2. 等待间隔从 80ms 降到 50ms
3. 用 `d(textContains="¥").exists` 替代完整 dump（~50ms vs ~500ms）

```python
# 优化前
for _ in range(5):
    self._cached_xml = self.d.dump_hierarchy()
    if '¥' in self._cached_xml or '￥' in self._cached_xml:
        ...
        return
    time.sleep(0.08)

# 优化后
for _ in range(3):
    if self.d(textContains="¥").exists or self.d(textContains="￥").exists:
        self._cached_xml = self.d.dump_hierarchy()
        ...
        return
    time.sleep(0.05)
# 最终兜底：只 dump 一次
self._cached_xml = self.d.dump_hierarchy()
```

**风险**: 低。textContains 比 dump_hierarchy 快 10 倍。

---

### P3: `_select_price_item()` 减少点击后等待（收益 30-80ms）

**文件**: `src/phase_machine.py` 第 371 行

当前：`time.sleep(0.08)` — 80ms
优化：`time.sleep(0.05)` — 50ms 或直接去掉

**风险**: 低。点击票档后页面跳转不需要 80ms 稳定时间。

---

### P4: `_find_price_coords()` 缓存 window_size()（收益 50ms）

**文件**: `src/phase_machine.py` 第 394 行

当前：每次调用 `self.d.window_size()` — USB 往返 ~50ms
优化：在 `__init__` 中缓存，或在 `_run_init` 中获取一次

```python
# 在 __init__ 中添加
self._screen_w, self._screen_h = None, None

# 在 _find_price_coords 中
if self._screen_w is None:
    self._screen_w, self._screen_h = self.d.window_size()
screen_w, screen_h = self._screen_w, self._screen_h
```

**风险**: 无。屏幕尺寸不会变化。

---

### P5: `_poll_until_activity_change()` 优化 poll 间隔（收益 20-50ms）

**文件**: `src/phase_machine.py` 第 224 行

当前：`poll_interval_ms=30`，但 dumpsys 调用本身需要 ~50ms
优化：`poll_interval_ms=50`，减少无效轮询

**风险**: 无。dumpsys 调用时间 > 30ms，当前间隔实际上没有等待。

---

### P6: `_ensure_price_page()` 场次选择后等待（收益 50-100ms）

**文件**: `src/phase_machine.py` 第 296、311 行

当前：`time.sleep(0.15)` — 150ms
优化：`time.sleep(0.1)` — 100ms

**风险**: 低。场次选择后页面加载不需要 150ms。

## 不建议优化的项

| 项 | 原因 |
|---|---|
| CPU 自旋等待开售 | 已是最优方案，精度毫秒级 |
| `_measure_rtt()` 单次采样 | 已优化过，N=2 中位数无统计意义 |
| sprint.py 页面检测间隔 | 用 d.info（~10ms）而非 dump_hierarchy，已是最优 |
| coord_blind.py blind_click | 已是 5ms 直接坐标点击，无法再快 |
| actions/ 模块的 sleep | 用于页面加载等待，属于必要等待 |

## 实施顺序建议

1. **先做 P0 + P1**（CONFIRM 阶段）：收益最大，改动集中在一个方法
2. **再做 P2**（ensure_price_page）：收益次大，需要改 dump → textContains 逻辑
3. **最后做 P3-P6**：收益较小，属于微调

1. 在探测模式下运行，对比优化前后日志中的耗时数据
2. 关注 `阶段转换` 日志的时间戳差值
3. 重点验证 CONFIRM 阶段的 `查找耗时` 字段
